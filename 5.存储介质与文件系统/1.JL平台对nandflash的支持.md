# 1. JL平台对NAND Flash的支持

## 1. 能力边界总览

### 1.1 JL已提供的能力（开箱即用）

| 能力项 | 说明 | 实现形式 |
|--------|------|----------|
| SPI NAND裸驱动 | 页读写、块擦除、坏块检测、ECC纠错 | 源码 `nandflash.c`（`nandflash_dev_ops`） |
| FTL闪存转换层 | 磨损均衡、逻辑-物理地址映射、坏块管理 | 源码 `ftl_device.c`（`ftl_dev_ops`）+ 预编译库 `ftl.a`（FTL 算法核心） |
| VFS文件系统挂载 | 将FTL设备挂载为FAT12/16/32/exFAT | 框架代码（源码可见） |
| USB Mass Storage | NAND Flash可以作为U盘暴露给Host | 框架代码（源码可见） |
| OTA固件升级 | 支持从NAND Flash读取升级文件 | 框架代码（源码可见） |
| 设备管理器集成 | 支持热插拔检测、mount/unmount管理 | 框架代码（源码可见） |

**一句话**：NAND 时序/ECC 在源码 `nandflash.c` 中，FTL 算法核心在预编译库 `ftl.a` 中；当前 earphone 工程需修改构建系统才能编译 NAND 支持，之后应用层即可通过 VFS 标准文件接口读写。

### 1.2 JL不支持/未开放的能力

| 能力项 | 说明 |
|--------|------|
| NAND驱动源码 | NAND 裸驱动 `nandflash.c` 和 FTL 适配层 `ftl_device.c` 均为厂商源码可见，但当前 earphone 工程默认不参与编译（`genFileList.c` 中仅在 `CONFIG_SOUNDBOX_CASE` 下编译） |
| FTL算法定制 | 无法调整磨损均衡策略、坏块管理阈值等FTL内部参数 |
| NAND专用文件系统 | 不像NOR Flash有 `nor_sdfile`（原生资源文件系统），NAND只支持通过FTL挂载FAT |
| SFC接口 | NAND Flash仅支持标准SPI接口，不像NOR有SFC（SPI Flash Controller）高速接口变体 |
| 多分区 | 当前设备注册中NAND FTL为单分区（`start_addr=0, size=128MB`），无内置多分区机制 |

### 1.3 NAND Flash vs NOR Flash 能力对比

| 维度 | NOR Flash | NAND Flash |
|------|-----------|------------|
| 裸驱动源码 | ✅ 可见（`norflash.c`, `norflash_sfc.c`） | ✅ 可见（`nandflash.c`） |
| FTL层 | ❌ 无FTL（NOR不需要） | ✅ 有FTL（`ftl_dev_ops`） |
| SPI接口 | ✅ 支持 | ✅ 支持 |
| SFC高速接口 | ✅ 支持（`norflash_sfc_dev_ops`） | ❌ 不支持 |
| 文件系统类型 | `nor_sdfile` / `rec_fs` / `fat` / `sdfile_fat` | `fat`（仅此一种） |
| 资源文件系统 | ✅ `nor_sdfile`（直接按文件名寻址） | ❌ 不支持 |
| USB U盘模式 | ❌ 不支持 | ✅ 支持 |
| OTA升级 | ✅ 支持（`USER_NORFLASH_UFW_UPDATA`） | ✅ 支持（`USER_NANDFLASH_UFW_UPDATA`） |

**关键差异总结**：NOR Flash在JL平台上有更丰富的文件系统生态（资源文件系统、录音文件系统、虚拟FAT），源码驱动可见。NAND Flash走的是"FTL + 标准FAT"的通用路线，裸驱动 `nandflash.c` 和 FTL 适配层 `ftl_device.c` 均为源码可见，适合需要大容量、文件交换（USB U盘）的场景。

### 2.1 设备注册模型

NAND Flash在JL平台采用**双设备注册**模式：

```c
// device_config.c (源码可见)
REGISTER_DEVICES(device_table) = {
#if TCFG_NANDFLASH_DEV_ENABLE
    {"nand_flash",      &nandflash_dev_ops, (void *)&nandflash_dev_data}, // 裸NAND设备
    {"nandflash_ftl",   &ftl_dev_ops,       NULL },                        // FTL抽象设备
#endif
};
```

- **`"nand_flash"`**：底层物理设备，由 `nandflash_dev_ops` 驱动，提供页级读写
- **`"nandflash_ftl"`**：FTL抽象设备，由 `ftl_dev_ops` 驱动，提供块设备接口给上层文件系统

文件系统挂载在 `"nandflash_ftl"` 而非 `"nand_flash"` 上。

### 2.2 设备管理器注册

```c
// dev_reg.c (源码可见)
#if (defined TCFG_NANDFLASH_DEV_ENABLE && TCFG_NANDFLASH_DEV_ENABLE)
{
    /*logo*/         "nandflash_ftl",
    /*name*/         "nandflash_ftl",
    /*storage_path*/ "storage/nandflash_ftl",
    /*root_path*/    "storage/nandflash_ftl/C/",
    /*fs_type*/      "fat",
},
#endif
```

设备管理器初始化时自动执行 `dev_manager_add("nandflash_ftl")` (`dev_manager.c:1084`)。

### 2.3 FTL 支持详解

**FTL (Flash Translation Layer)** 是 NAND Flash 与文件系统之间的关键抽象层。NAND Flash 存在两个固有问题：(1) 写入前必须先擦除，且擦除以块为单位；(2) 存在出厂坏块和使用中产生的新坏块。FTL 将这些复杂性隐藏，向上层呈现一个"逻辑块设备"。

#### 2.3.1 实现状态

| 维度 | 说明 |
|------|------|
| FTL 算法核心 | `cpu/br28/liba/ftl.a`（预编译库，**闭源**）：磨损均衡、垃圾回收、逻辑-物理映射 |
| FTL 适配层 | `apps/common/device/storage_device/nandflash/ftl_device.c`（**源码可见**）：定义 `ftl_dev_ops`，调用 `ftl.a` 核心 |
| 接口声明 | `nandflash.h` — `extern const struct device_operations ftl_dev_ops;` |
| 源码 | ⚠️ FTL 算法核心无源码；但 `ftl_device.c` 适配层可修改 |
| 配置项 | ❌ FTL 核心无公开配置参数；`ftl_device.c` 可通过 `nandflash_get_ftl_info()` 传递几何参数 |

FTL 算法核心是一个**黑盒**，但 `ftl_device.c` 适配层是源码可见的。开发者可以修改适配层来传递不同的 NAND 几何参数，但无法调整 `ftl.a` 内部的磨损均衡策略、GC 阈值等算法细节。

#### 2.3.2 在架构中的位置

```
fopen("storage/nandflash_ftl/C/file.txt", "w")
         │
         ▼
    VFS (fs.h)          ← 统一文件接口
         │
         ▼
    FAT 文件系统          ← 来自 fs.a (闭源)，管理目录/簇链/FAT表
         │
         ▼  通过 dev_io_t { read, write, ioctrl }
    FTL (ftl.a)          ← 闭源黑盒：逻辑块 → 物理块映射、磨损均衡、坏块管理
         │
         ▼  nand_flash_page_read / nand_flash_page_write / nand_flash_erase
    NAND 裸驱动 (nandflash.c) ← 源码可见：SPI时序、ECC纠错、坏块标记
         │
         ▼  SPI 总线
    NAND Flash 芯片
```

#### 2.3.3 FTL 通过 `dev_io_t` 暴露的接口

FTL 作为一个块设备，通过标准的 `dev_io_t` 结构体向上层文件系统提供读写接口。上层文件系统（FAT）无需感知 NAND 的页/块结构，只需按逻辑扇区号读写：

```c
// device_drive.h — dev_io_t 接口定义（源码可见）
typedef struct DEV_IO {
    const char name[8];
    s32(*mount)(void *volatile parm);    // 设备挂载
    s32(*unmount)();                     // 设备卸载
    s32(*read)(u8 *volatile buf, u32 addr, u32 len);   // 逻辑块读取
    s32(*write)(u8 *volatile buf, u32 addr, u32 len);  // 逻辑块写入
    s32(*ioctrl)(void *volatile parm, u32 cmd);         // IO控制（获取块大小、块数量、擦除等）
    s32(*power)(u32 mod);
    s32(*detect)();
    struct dev_mutex *mutex;
    dev_type device_type;               // DEV_NAND_FLASH
    void *private_data;
} dev_io_t;
```

通用 IOCTL 命令（所有存储设备支持）：

| 命令 | 说明 |
|------|------|
| `DEV_GET_STATUS` | 获取设备状态（在线/离线/休眠） |
| `DEV_GET_BLOCK_SIZE` | 获取逻辑块大小 |
| `DEV_GET_BLOCK_NUM` | 获取逻辑块总数 |
| `DEV_GET_TYPE` | 返回设备类型（`DEV_NAND_FLASH`） |
| `DEV_SECTOR_ERASE` | 扇区擦除 |
| `DEV_BLOCK_ERASE` | 块擦除 |
| `DEV_CHIP_ERASE` | 全片擦除 |

#### 2.3.4 操作流程

**挂载流程**（`dev_manager_add("nandflash_ftl")` 自动触发）：

```
mount("nandflash_ftl", "storage/nandflash_ftl", "fat", 3, NULL)
  → 通过设备名 "nandflash_ftl" 找到 ftl_dev_ops
  → 调用 FTL 的 mount() 初始化 FTL 内部状态（扫描坏块表、重建映射表等）
  → VFS 读取 FTL 设备的块大小/块数量
  → FAT 文件系统读取 super block / boot sector，验证文件系统类型
  → 返回 struct imount * 挂载句柄
```

**格式化流程**（需手动调用）：

```
f_format("nandflash_ftl", "fat", 4096)
  → 找到已注册的 FTL 设备
  → 通过 FTL 的 ioctrl 获取设备容量
  → FAT 格式化代码写入 boot sector、FAT 表、根目录
  → 返回 0 表示成功
```

### 2.4 文件系统支持详解

JL 平台的 VFS (Virtual File System) 框架是**源码完全可见**的，位于 `SDK/interface/utils/fs/fs.h`。但各具体文件系统的**底层实现大部分闭源**。

#### 2.4.1 VFS 框架架构

VFS 提供统一的文件操作接口，通过 `struct vfs_operations` 注册不同的文件系统实现：

```c
// fs.h — VFS 核心数据结构（源码可见）
struct vfs_operations {
    const char *fs_type;              // 文件系统类型名，如 "fat", "sdfile"
    int (*mount)(struct imount *, int);
    int (*unmount)(struct imount *);
    int (*fopen)(FILE *, const char *path, const char *mode);
    int (*fread)(FILE *, void *buf, u32 len);
    int (*fwrite)(FILE *, void *buf, u32 len);
    int (*fseek)(FILE *, int offset, int);
    int (*flen)(FILE *);
    int (*fpos)(FILE *);
    int (*fget_name)(FILE *, u8 *name, int len);
    int (*fget_path)(FILE *, struct vfscan *, u8 *name, int len, u8 is_relative_path);
    int (*fclose)(FILE *);
    int (*fdelete)(FILE *);
    int (*ioctl)(void *, int cmd, int arg);
    vfs_extern *ops_extern;          // 扩展操作：format, fscan, fsel, etc.
};
```

**挂载信息结构**：

```c
struct imount {
    int fd;
    const char *path;                // 挂载点路径
    struct vfs_operations *ops;      // 指向对应文件系统操作表
    struct vfs_devinfo dev;          // 底层设备信息（块大小、FAT基址等）
    struct vfs_partition part;       // 分区信息（偏移、簇大小、总容量）
    struct list_head entry;          // 链表节点
    atomic_t ref;
    OS_MUTEX mutex;
    u8 avaliable;
    u8 part_num;
};
```

#### 2.4.2 JL 支持的文件系统类型

从 `dev_reg.c` 和各头文件可归纳出 JL 平台支持的所有文件系统：

| fs_type 字符串 | 用途 | 存储介质 | 实现可见性 | 说明 |
|---------------|------|---------|-----------|------|
| `"fat"` | 通用 FAT 文件系统 | SD卡、U盘、NOR FAT、**NAND FTL** | ❌ `fs.a` 闭源 | 支持 FAT12/16/32/exFAT，标准文件系统 |
| `"sdfile"` | 内置 Flash 资源文件系统 | 芯片内置 Flash | ❌ 闭源 | 轻量级读优化文件系统，用于固件资源（音频、配置等） |
| `"nor_sdfile"` | 外挂 NOR 资源文件系统 | 外挂 NOR Flash | ❌ 闭源 | 与 `sdfile` 同格式，用于外挂 NOR 上的资源文件 |
| `"sdfile_fat"` | 虚拟 FAT 文件系统 | 外挂 Flash（模拟FAT） | ✅ **源码可见** | `virfat_flash.c`，将 sdfile 格式资源映射为 FAT 目录结构，USB 可读 |
| `"rec_fs"` | 录音文件系统 | 外挂 NOR Flash | ❌ 闭源 | 轻量级顺序写文件系统，用于录音场景 |
| `"rec_sdfile"` | 内置 Flash 录音文件系统 | 芯片内置 Flash | ❌ 闭源 | 内置 Flash 上的录音专用文件系统 |

#### 2.4.3 各文件系统适用场景

```
                         ┌─ 芯片内置Flash ── "sdfile" (资源文件，只读优化)
                         │                 ── "rec_sdfile" (录音专用)
                         │
存储介质 ──┼─ SD卡 / U盘 ──── "fat" (FAT12/16/32/exFAT)
                         │
                         ├─ 外挂 NOR Flash ── "nor_sdfile" (资源文件)
                         │                  ── "rec_fs" (录音)
                         │                  ── "fat" (通用文件，可被PC识别)
                         │                  ── "sdfile_fat" (虚拟FAT，USB映射)
                         │
                         └─ 外挂 NAND Flash ── "fat" (唯一选择，通过FTL)
```

#### 2.4.4 NAND Flash 的文件系统选择

**NAND Flash 只支持 `"fat"` 这一种文件系统类型**。原因：

1. NAND 的物理特性（坏块、需擦除）决定了必须通过 FTL 层抽象
2. FTL 暴露的是标准块设备接口，天然适合 FAT 这类通用文件系统
3. `sdfile`/`nor_sdfile` 等轻量文件系统是直接操作 NOR Flash 地址的，不经过 FTL，无法在 NAND 上使用

```c
// dev_reg.c — NAND FTL 设备只能配 "fat"
{
    /*logo*/         "nandflash_ftl",
    /*name*/         "nandflash_ftl",
    /*storage_path*/ "storage/nandflash_ftl",
    /*root_path*/    "storage/nandflash_ftl/C/",
    /*fs_type*/      "fat",              // ← 只能是 "fat"
},
```

#### 2.4.7 与 NOR Flash 文件系统的关键差异

对于 NAND Flash 开发者，需要了解 NOR Flash 上的专用文件系统在 NAND 上**不可用**：

- **`sdfile` / `nor_sdfile`**：这是 JL 自有的轻量文件系统，文件直接按名字索引存储，没有目录树/FAT 表的概念。它直接操作 Flash 物理地址，不经过 FTL，因此**无法在 NAND 上使用**。格式由 PC 端工具 `fat_comm.exe` 打包生成。
- **`rec_fs`**：专为录音场景优化的顺序写文件系统，直接管理 NOR Flash 扇区，**无法在 NAND 上使用**。
- **`sdfile_fat`**（`virfat_flash.c`）：这是本 repo 中**唯一有完整源码的文件系统实现**。它的作用是将 NOR Flash 上的 sdfile 格式资源文件"伪装"成 FAT 目录结构，使得通过 USB 连接 PC 时能被识别为 U 盘。

#### 不同厂商 NAND Flash 的适配

`nandflash.c` 是源码可见的厂商驱动，内部通过 `Read ID` 命令读取芯片 ID，然后在 switch-case 中查表匹配芯片参数（类似 `norflash.c` 的 `_norflash_read_id()` 机制）。

**对开发者的影响：**

| 场景 | 处理方式 |
|------|---------|
| 使用已支持芯片（如 XT26G01C 等） | 开箱即用，驱动内已有 case。只需正确配置 SPI 引脚和线宽 |
| 使用新型号芯片（如 GD5F4GM7UE） | **可以自行适配**：在 `nandflash.c` 的 switch(read_id) 中新增 case，配置 page_size、block_size、ECC 参数等；同步修改 `ftl_device.c` 解除硬编码 |
| 需要查看支持的芯片列表 | 直接查看 `nandflash.c` 中的 switch-case 即可 |

**关键认知**：NAND 裸驱动不是黑盒。开发者可以像修改 NOR 驱动一样修改 `nandflash.c` 来新增芯片支持。但需注意：
1. `ftl.a` 中的 FTL 算法核心是闭源的，它期望裸驱动通过标准接口（`page_read`/`page_write`/`erase_block`）正确工作
2. 新增芯片时必须确保几何参数（page_size、block_size、pages/block）与 datasheet 一致，否则 FTL 可能工作异常

> **⚠️ 重要限制**：虽然裸驱动源码可见，但 `ftl.a` 是闭源黑盒。如果新芯片的物理特性（如 OOB 布局、ECC 机制）与 `ftl.a` 内部假设严重不兼容，则可能需要联系 JL 评估 `ftl.a` 的兼容性。

#### SPI 传输模式与线宽说明

**关于全双工 vs 半双工的实际情况**

BR28 SPI 控制器硬件上定义了 4 种模式（`interface/driver/cpu/periph/spi.h`），但对于 **SPI Flash/NAND 应用**，全双工和半双工在实际使用中没有区别——因为 Flash 从机不会主动发数据，始终是主机先发命令/地址，从机再响应数据。

> 实测验证：使用 `SPI_MODE_UNIDIR_1BIT`（半双工）成功初始化并读取了 GD5F4GM7 的 Flash ID，与全双工模式效果相同。

**因此，开发者无需纠结全双工/半双工，只需关注线宽配置。**

**NAND Flash 支持的读数据线宽**（由 `spi_read_width` 参数控制）：

参考 GD5F4GM7 命令集，数据读取阶段支持以下模式：

| `spi_read_width` | 读命令 | 数据线 | 速度 | 说明 |
|------------------|--------|--------|------|------|
| `1` | `03h` / `0Bh` | SI(发) + SO(收) | 基准 | Standard SPI，兼容性最好 |
| `2` | `3Bh` (x2) / `BBh` (Dual IO) | SIO0 + SIO1 | x2 | Dual SPI，两线并行输出 |
| `4` | `6Bh` (x4) / `EBh` (Quad IO) | SIO0~SIO3 | x4 | Quad SPI，需设置 QE=1 |

**传输时序：**

```
┌──────────────────────────────────────────────────────────────┐
│                    SPI NAND Flash 传输时序                     │
├──────────────┬──────────────┬────────────────────────────────┤
│   命令阶段    │   地址阶段    │           数据读取阶段           │
├──────────────┼──────────────┼────────────────────────────────┤
│ Standard SPI  │ Standard SPI │ 由 spi_read_width 决定：        │
│ SI→发命令    │ SI→发地址    │  • 1: Standard SPI (SO接收)    │
│              │              │  • 2: Dual SPI (SIO0+SIO1)     │
│              │              │  • 4: Quad SPI (SIO0~SIO3)     │
└──────────────┴──────────────┴────────────────────────────────┘
```

- 命令和地址阶段始终用 **Standard SPI**（SI 发送）
- 数据读取阶段根据 `spi_read_width` 切换线宽
- 写数据（Program Load）的线宽由驱动内部决定，应用层不可配

**配置示例：**

```c
// 保守配置：Standard SPI 读，兼容性最好
#define TCFG_FLASH_DEV_FLASH_READ_WIDTH  1

// 平衡配置：Dual SPI 读，速度翻倍
#define TCFG_FLASH_DEV_FLASH_READ_WIDTH  2

// 性能配置：Quad SPI 读，速度 x4（需芯片支持 QE）
#define TCFG_FLASH_DEV_FLASH_READ_WIDTH  4
```

**关键限制：**

| 限制项 | 说明 |
|--------|------|
| `spi_read_width` **仅控制数据读阶段** | 命令和地址始终用 Standard SPI |
| 写数据（Program Load）线宽不可配置 | 由驱动内部决定 |
| SPI 时钟相位/极性（CPOL/CPHA）不可配置 | 由驱动内部管理 |
| 仅支持 SPI1 / SPI2 | 不支持 SPI0 |

#### 单线半双工硬件设计（已验证通过）

**设计动机：** 标准 SPI 双线方案（SI + SO）在主控侧占用了 **2 个 GPIO**。对于 TWS 耳机/充电仓等引脚极度紧张的微型设备，每省一个 IO 都意味着可以多放一个功能。

**硬件方案：单线 Composite 双向数据线**

```
MCU (SPI_HOST)                       GD5F4GM7 (SPI_NAND)
┌──────────────┐                     ┌──────────────┐
│              │                     │              │
│    SDO ──────┼──────┬──────────────┼── SI (引脚5)  │
│              │      R (330Ω)       │              │
│    SDI ──────┼──────┘              │              │
│              │      │              │              │
│              │      └──────────────┼── SO (引脚2)  │
│              │                     │              │
│    SCLK ─────┼────────────────────┼── SCLK        │
│     CS# ─────┼────────────────────┼── CS#         │
└──────────────┘                     └──────────────┘
```

用一颗 **330Ω 贴片电阻**，将 Flash 的 SI/SO 两根引脚在物理上合为一根 Data 线接到 MCU。MCU 侧只占用 **1 个 GPIO**。

**工作原理 — 分时半双工：**

| 阶段 | MCU GPIO 方向 | Flash SI 状态 | Flash SO 状态 | 数据流向 |
|------|-------------|-------------|-------------|---------|
| 发命令/地址 | 输出（驱动电平） | 监听 | **高阻态 (Hi-Z)** | MCU → Flash |
| 写数据 | 输出（驱动电平） | 监听 | **高阻态 (Hi-Z)** | MCU → Flash |
| 读数据 | 输入（不驱动） | 忽略 | **驱动电平** | Flash → MCU |

Flash 的 SO 引脚在非读阶段天然处于高阻态，不会干扰 MCU 发送信号。330Ω 电阻的作用是：在 MCU 和 Flash 方向切换的**极短瞬间**，如果双方同时驱动不同电平，电阻限制短路电流，保护引脚不被烧毁。

**与手册推荐的标准双线方案的对比：**

| 维度 | 手册推荐（双线 Standard SPI） | 本方案（单线 Composite） |
|------|---------------------------|----------------------|
| MCU 占用 IO | **2 个**（SI + SO） | **1 个** |
| Flash 物理引脚 | SI(引脚5) + SO(引脚2) | 不变，仍然 SI + SO 都连出 |
| 外部元件 | 无 | 1 颗 330Ω 电阻（成本 < ¥0.001） |
| 通信模式 | `SPI_MODE_BIDIR_1BIT`（全双工） | `SPI_MODE_UNIDIR_1BIT`（半双工） |
| 实际吞吐 | 命令→地址→数据，天然半双工 | 与双线完全相同 |
| Dual/Quad SPI 多线读 | ✅ 支持（SI+SO 变为 SIO0+SIO1） | ❌ 不支持（两根线已被电阻桥接） |
| 驱动兼容性 | ✅ 所有 SPI 驱动通用 | ⚠️ 需确认 HAL 层支持半双工切换（已实测通过） |
| 适用场景 | IO 充裕的标准板 | IO 极度紧张的微型设备（TWS 充电仓、手环等） |

**⚠️ 牺牲的能力：** 因为 SI 和 SO 被电阻短接，Dual SPI (x2) 和 Quad SPI (x4) 多线读取无法使用。`spi_read_width` 必须固定为 `1`。

**实测验证：** 已在 BR28 平台上使用 `SPI_MODE_UNIDIR_1BIT`（半双工）+ 单线 Composite 硬件设计，成功初始化 GD5F4GM7 并读取 Flash ID，读写正常。

**配置要点：**

```c
// 板级配置：单线半双工
#define TCFG_FLASH_DEV_FLASH_READ_WIDTH  1   // 必须为 1，多线读已不可用

// device_config.c：SPI 平台数据
// .port[1] (DO) = Data 线
// .port[2] (DI) = 与 DO 同一个 IO（或填 -1）
// .mode        = SPI_MODE_UNIDIR_1BIT   // 半双工，DO 分时收发
```

**关键认知：驱动代码写法不变，只改硬件配置。**

即使硬件上是单线 Composite 设计，写驱动时仍然遵循标准 SPI 的 SO/SI 逻辑：

```
spi_write_byte(cmd);    // 发命令  → SPI 控制器自动切 DO 为输出，驱动 Data 线
spi_write_byte(addr);   // 发地址  → 同上
data = spi_recv_byte(); // 收数据  → SPI 控制器自动切 DO 为输入，读取 Data 线
```

SPI 控制器的 `SPI_MODE_UNIDIR_1BIT`（半双工）模式在硬件层面完成了方向切换——DO 引脚在 `send` 时是输出、在 `recv` 时自动变为输入。上层驱动代码**完全不用感知**物理上是两根线还是一根线——只要 SPI 平台数据配置中 mode 正确、DO/DI 指向同一个 IO，控制器的半双工硬件就自动处理方向切换。

这一点尤为重要：即使 `nandflash.c` 是源码可见的，其内部也调用标准的 `spi_write_byte` / `spi_recv_byte` 等 HAL 接口。只要 SPI 硬件配置正确（半双工模式 + DO/DI 指向同一个 IO），这些 HAL 调用就能在半双工单线上正常工作，无需修改 NAND 驱动内部逻辑。

## 5. 常见问题

### Q1: NAND Flash 和 NOR Flash 可以同时使用吗？

可以。两者使用不同的 SPI 控制器或不同的 CS 片选引脚即可。在板级配置中同时使能 `TCFG_NANDFLASH_DEV_ENABLE` 和 `TCFG_NORFLASH_DEV_ENABLE`（或相关的 NOR 配置宏）。

### Q2: NAND Flash 是否需要额外的坏块管理？

不需要。FTL 层已经内置了坏块管理，应用层通过 FAT 文件系统读写时完全透明。

### Q3: 是否需要自己写驱动？

**NAND 裸驱动不需要从零编写**，但可能需要自行适配。`nandflash.c`（裸驱动）和 `ftl_device.c`（FTL 适配层）是厂商源码，可以通过修改源码来适配新芯片：

- 在 `nandflash.c` 的 switch(read_id) 中新增芯片 case，配置 page_size、block_size、ECC 参数等
- 在 `ftl_device.c` 中通过 `nandflash_get_ftl_info()` 获取参数，无需硬编码特定芯片

> **注意**：`ftl.a` 中的 FTL 算法核心确实是闭源黑盒，但其接口是标准化的。只要裸驱动能正确读写页/块，`ftl.a` 通常可以透明工作。

应用层只需要：
1. 修改构建系统使 earphone 工程编译 `nandflash.c` + `ftl_device.c`
2. 使能宏并配置 SPI 引脚
3. 格式化
4. 通过 VFS 标准接口读写
