## 一、存储介质分类

### 1.1 存储介质的两大阵营

#### 1.1.1 第一类：易失性存储 (Volatile Memory)

特点：读写速度极快，但断电后数据立即丢失。

- **SRAM (静态随机存储器)：**
  - **形式：** 绝大多数 SRAM 集成在 SoC/MCU 内部（用作 Cache 或临时寄存器），少数场景下会外挂独立 SRAM 芯片（如 ISSI、Cypress 等厂商产品，用于需要大容量高速缓存的场景）。
  - **角色：** 缓存（Cache）或临时寄存器。
- **DRAM (动态随机存储器)：**
  - **形式：** PC 上为内存条；手机或嵌入式系统中为焊接在主板上的独立 LPDDR 颗粒。
  - **角色：** 系统运行时的主内存与工作台。

#### 1.1.2 第二类：非易失性存储 (Non-Volatile Memory / Flash)

特点：读写速度相对较慢，但断电后数据可持久保存。

- **NOR Flash：**
  - **形式：** 小容量可内置在单片机中，大容量以外挂形式存在于主板上（通常通过 SPI 接口连接）。
  - **角色：** 存储启动指令，支持直接执行代码 (XIP, eXecute In Place)。
- **SPI NAND / Parallel NAND：**
  - **形式：** 独立的外挂芯片。
  - **角色：** 性价比高的中等容量存储，用于存放固件和用户数据。
- **eMMC / SSD / UFS：**
  - **形式：** 高度集成的存储模块（包含 Flash 颗粒和内部控制器）。
  - **角色：** 大容量存储仓库，用于存储操作系统、应用程序、视频、照片等。

### 1.2 存在形式的本质差异

存储介质在工程上的差异主要体现在**接口（连接方式）**和**管理逻辑**两个维度。

#### 1.2.1 连接方式：并行 vs 串行

- **DRAM（内存条）：** 必须使用并行总线（引脚数量多且密集，如 DDR4/DDR5），因为速度要求极高，需要多车道并行传输数据。
- **SPI Flash（NOR/NAND）：** 常用串行总线（通常仅 4~8 个引脚），节省 PCB 空间和主控引脚，虽然速度较慢但在启动场景下足够使用。
- **eMMC：** 介于两者之间，通常使用 8 位并行接口。
- **UFS：** 使用 MIPI M-PHY 物理层，串行全双工接口，带宽远超 eMMC。

#### 1.2.2 芯片内部：裸颗粒 vs 带控制器

- **NOR/NAND Flash：** 大多是"裸颗粒"。主控芯片（CPU）必须直接处理纠错码（ECC）、坏块管理等底层操作。
- **eMMC / SSD / UFS：** 内部自带一个微型控制器（Flash 主控）。主控只需发送高级指令如"存这个文件"，内部控制器自行处理复杂的 NAND 物理特性，CPU 无需操心。

### 1.3 设备中的存储链条

以一台正在运行的智能相机为例：

1. **外挂 NOR Flash** 存储 Bootloader（开机第一句话）。
2. **eMMC** 存储 Linux 操作系统和录像视频（大仓库）。
3. **DRAM (内存条)** 运行当前的图像处理算法（工作台）。
4. **CPU 内部的 SRAM** 正在计算当前这一帧画面的曝光数值（高速运算单元）。

## 二、NAND Flash 技术概述

**NAND Flash 是一种存储介质/技术类型的统称**，类似于"硬盘"或"内存"这样的类别名称，而非某个厂商的专属产品。任何具备相应技术能力的半导体厂商都可以设计和生产 NAND Flash 芯片。

### 2.1 主要厂商

| 类型             | 代表厂商                                                     |
| :--------------- | :----------------------------------------------------------- |
| **国际大厂**     | 三星(Samsung)、铠侠(Kioxia，原东芝)、西部数据/闪迪(WD/SanDisk)、美光(Micron)、SK海力士(SK hynix)、英特尔(Intel，已出售给SK海力士) |
| **中国大陆厂商** | 长江存储(YMTC，NAND厂商代表)、兆易创新(GigaDevice，GD5F4GM7 即为其产品) |
| **其他厂商**     | 旺宏(Macronix)、华邦(Winbond)、长鑫存储(CXMT，主要做DRAM) 等 |

### 2.2 SLC/MLC/TLC/QLC 对比

NAND Flash 根据每个存储单元存储的位数不同，分为四种类型：

| 类型 | 每单元位数 | P/E 寿命 | 速度 | 成本 | 应用 |
|------|-----------|---------|------|------|------|
| **SLC** | 1 | ~80K | 最快 | 最高 | 工业级、嵌入式 |
| **MLC** | 2 | ~3K-10K | 快 | 高 | 企业级 SSD |
| **TLC** | 3 | ~1K-3K | 中 | 中 | 消费级 SSD、手机 |
| **QLC** | 4 | ~几百 | 慢 | 最低 | 大容量存储 |

> **核心权衡**：每单元存储的位数越多，容量越大、成本越低，但速度和寿命也随之下降。嵌入式系统通常选用 SLC 或 MLC 以获得更高的可靠性。

### 2.3 GD5F4GM7 芯片介绍

**GD5F4GM7** 是 **兆易创新(GigaDevice)** 推出的 **SPI-NAND Flash** 产品：

- 容量：4Gb（512MB）
- 接口：SPI 串行接口（区别于传统的并行 NAND）
- 特点：内置 ECC、支持 Quad SPI/DTR 等高速模式
- 子型号区分：
  - **xE 型号**：2KB+128B Page / 128KB+8KB Block
  - **xF 型号**：4KB+256B Page / 256KB+16KB Block

SPI-NAND 是 NAND Flash 的一个细分品类，主打**引脚少、成本低、易于集成**，常用于嵌入式系统、IoT 设备、路由器、机顶盒等场景，作为 SPI-NOR Flash 的高密度替代方案。

### 2.4 类比说明

| 类比                                                         | 说明                 |
| :----------------------------------------------------------- | :------------------- |
| NAND Flash ≈ "轿车"                                          | 是一类产品的统称     |
| 三星/铠侠/长江存储/兆易创新 ≈ "丰田/大众/比亚迪/吉利"        | 是具体的厂商品牌     |
| SLC/MLC/TLC/QLC 或 SPI-NAND/Raw NAND ≈ "燃油车/混动/纯电" 或 "轿车/SUV" | 是技术路线和细分品类 |

所以不同厂商的 NAND Flash 在**底层存储原理**上是相通的（都基于浮栅晶体管的电荷存储），但在**具体工艺制程、接口标准、可靠性指标、价格定位**等方面会有差异。

### 2.5 NAND Flash 接口分类

**NAND Flash ≠ SPI NAND Flash**，SPI 只是 NAND Flash 的**其中一种接口类型**，而非唯一选择。

| 接口类型                 | 全称                             | 特点                                                         | 典型应用场景                                    |
| :----------------------- | :------------------------------- | :----------------------------------------------------------- | :---------------------------------------------- |
| **Raw NAND / 并行 NAND** | Parallel NAND / ONFI NAND        | 8-bit 或 16-bit 并行数据总线，引脚多（30+），速度快，需要复杂控制器 | 大容量存储：SSD、手机 eMMC/UFS、SD卡            |
| **SPI NAND**             | Serial Peripheral Interface NAND | 4线串行（SCLK+CS+SI+SO），引脚少（8个），简单易用，速度较慢  | 嵌入式系统：路由器、机顶盒、IoT 设备、BIOS 存储 |
| **Toggle NAND**          | Toggle Mode NAND                 | 基于差分信号的异步接口，由东芝/三星主导，速度快于传统并行    | 高性能 SSD、企业级存储                          |
| **UFS**                  | Universal Flash Storage          | 独立命令/数据通道、全双工、串行接口、带宽远超 eMMC           | 现代智能手机、高端平板（UFS 2.x/3.x/4.x）       |

#### 2.5.1 UFS 与 eMMC 核心区别

- **eMMC** 使用 8 位并行接口，半双工
- **UFS** 使用 MIPI M-PHY 物理层，全双工，命令和数据可并行传输
- **UFS 3.1** 顺序读可达 2100MB/s，远超 eMMC 5.1 的 400MB/s
- 当前智能手机已基本全面转向 UFS

### 2.6 SPI NAND 与 Raw NAND 的核心区别

| 对比项          | Raw NAND (并行)             | SPI NAND (串行)                                          |
| :-------------- | :-------------------------- | :------------------------------------------------------- |
| **引脚数量**    | 30+ 个                      | 8 个（含 VCC/GND）                                       |
| **接口复杂度**  | 复杂，需专用 NAND 控制器    | 简单，标准 SPI 控制器即可                                |
| **传输速度**    | 快（几十 MB/s ~ 几百 MB/s） | 较慢（标准 SPI 约几 MB/s，Quad SPI 可到几十 MB/s）       |
| **ECC 要求**    | 通常由外部控制器处理        | 常内置 ECC（如 GD5F4GM7 内置了 8bit/528B ECC） |
| **容量范围**    | 大（几 GB ~ 几 TB）         | 较小（128Mb ~ 32Gb 常见）                                |
| **成本/易用性** | 需要额外控制器，整体成本高  | 芯片本身成本低，集成简单                                 |

### 2.7 GD5F4GM7 的分类定位

```
GD5F4GM7 = SPI NAND Flash
         ↓
         4Gb 容量
         8 个引脚（WSON8 封装）
         SPI 串行接口（支持 Standard/Dual/Quad/DTR 模式）
         内置 ECC 引擎
         xE 型号：2KB+128B Page / 128KB+8KB Block
         xF 型号：4KB+256B Page / 256KB+16KB Block
         目标：替代 SPI-NOR，提供更高密度、更低成本的嵌入式存储
```

### 2.8 小结

> **NAND Flash** 是**存储介质技术**的统称（基于浮栅晶体管）
>
> **SPI NAND Flash** 是 NAND Flash 的**一个子类别**，强调使用 **SPI 串行接口** 来简化与主控的连接
>
> 此外还有 **Raw NAND**、**Toggle NAND**、**UFS** 等其他接口类型的 NAND Flash，它们底层存储原理相同，但对外接口和适用场景完全不同。

**SPI 只是 NAND Flash 的可选接口之一，而非 NAND Flash 的固有属性。**

## 三、NOR Flash 与 NAND Flash 在文件系统层面的差异

**存储介质对文件系统的"兼容性"不是无条件的。**

### 3.1 物理特性差异导致文件系统选择不同

| 特性            | NOR Flash                | NAND Flash                   |
| :-------------- | :----------------------- | :--------------------------- |
| **读取粒度**    | 字节级随机读（像 RAM）   | 页级顺序读（必须整页读）     |
| **写入粒度**    | 按字/字节编程            | 必须页编程（最小 2KB/4KB）   |
| **擦除粒度**    | 扇区擦除（常见 64KB 或 4KB sub-sector，具体因厂商和型号而异） | 块擦除（128KB/256KB 典型）   |
| **坏块**        | 出厂无坏块，寿命期内极少 | 出厂即有坏块，使用中持续产生 |
| **位翻转**      | 极少                     | 需要 ECC 纠正                |
| **寿命（P/E）** | 10万~100万次             | SLC 约 80K，MLC 约 3K-10K，TLC 约 1K-3K，QLC 仅几百次 |
| **成本/密度**   | 贵、低密度（MB 级）      | 便宜、高密度（GB 级）        |

### 3.2 NOR Flash 使用场景

因为 NOR 可以**字节随机读、直接执行代码（XIP）**，所以：

| 典型用途               | 文件系统/使用方式          | 原因                    |
| :--------------------- | :------------------------- | :---------------------- |
| 固件存储（Bootloader） | **不用文件系统**，直接裸跑 | 需要 CPU 直接取指令执行 |
| 小配置数据             | **JFFS2 / 裸数据**         | 简单、可写少            |
| 大容量存储             | **几乎不用 NOR**           | 成本太高                |

**NOR 通常不需要 FTL**，因为它没有坏块问题，擦写寿命容忍度高。

### 3.3 NAND Flash 使用场景

NAND 的物理特性决定了必须配合特定机制：

| NAND 问题      | 必须的处理          | 对文件系统的要求                    |
| :------------- | :------------------ | :---------------------------------- |
| 出厂坏块       | 跳过坏块，用备用块  | 文件系统要知道"逻辑地址≠物理地址"   |
| 写入前必须擦除 | 异地更新 + 垃圾回收 | 文件系统不能原地改写元数据          |
| 擦写次数有限   | 磨损均衡            | 不能总写同一个物理区域              |
| 位翻转         | ECC 校验            | 文件系统需预留 OOB/spare 空间给 ECC |

### 3.4 文件系统的介质感知程度

**不是所有文件系统都能随便用在所有介质上。** 文件系统分为两类：

#### 3.4.1 块设备通用文件系统（介质无关）

| 文件系统           | 依赖的底层                  | 能否用于裸 NAND           |
| :----------------- | :-------------------------- | :------------------------ |
| **FAT16/32/exFAT** | 标准块设备（512B/4KB 扇区） | ❌ **不能直接用于裸 NAND** |
| **NTFS**           | 标准块设备                  | ❌                         |
| **ext2/3/4**       | 标准块设备                  | ❌                         |

**为什么 U 盘可以格式化成 FAT/exFAT/NTFS？**

因为 U 盘内部有 **USB 控制器 + FTL**，对外呈现的是**标准块设备**（SCSI/USB Mass Storage 协议）。PC 看到的不是原始 NAND，而是"虚拟硬盘"：

```
PC 端：格式化工具
    ↓  USB Mass Storage 协议
U 盘控制器：接收块地址（LBA 0, 1, 2...）
    ↓  内部 FTL 转换
NAND 芯片：物理页/块操作
```

**FTL 隐藏了 NAND 的所有物理特性**，让上层以为是在操作"完美块设备"。所以 FAT 可以工作——FAT 不知道也不关心底层是 NAND、SSD 还是机械硬盘。

#### 3.4.2 裸 Flash 专用文件系统（介质强相关）

这些文件系统**直接操作物理/逻辑页、块，不经过标准块设备抽象**：

| 文件系统     | 设计目标                | 特点                             |
| :----------- | :---------------------- | :------------------------------- |
| **JFFS2**    | 裸 NOR/NAND             | 日志结构、原地压缩、磨损均衡内置 |
| **YAFFS2**   | 裸 NAND（大页）         | 每页一个文件头、启动快、无压缩   |
| **UBIFS**    | 裸 NAND（配合 UBI 层）  | 现代替代 JFFS2，支持动态卷       |
| **LittleFS** | 裸 NOR/NAND（MCU 场景） | 极简、掉电安全、磨损均衡         |

**这些文件系统自带"类 FTL"功能**——它们自己管理坏块、磨损均衡、日志回滚。

### 3.5 文件系统与介质的匹配关系

```
┌─────────────────────────────────────────┐
│  上层应用                               │
├─────────────────────────────────────────┤
│  文件系统层                              │
│                                         │
│  ├─ 块设备通用 FS (FAT/NTFS/ext4)        │
│  │   └── 要求底层：标准块设备（LBA 寻址）  │
│  │   └── 必须配合：FTL（在芯片内或驱动中） │
│  │                                        │
│  └─ 裸 Flash 专用 FS (JFFS2/YAFFS2/UBIFS) │
│      └── 要求底层：原始 Flash 接口        │
│      └── 自带：坏块管理、磨损均衡、ECC    │
├─────────────────────────────────────────┤
│  中间层                                  │
│  ├─ FTL（可选）→ 块设备抽象               │
│  └─ UBI（UBIFS 需要）→ 卷管理             │
├─────────────────────────────────────────┤
│  驱动层                                  │
├─────────────────────────────────────────┤
│  物理介质：NOR / NAND / eMMC / SSD / HDD  │
└─────────────────────────────────────────┘
```

| 组合                       | 是否可行 | 例子                      |
| :------------------------- | :------- | :------------------------ |
| FAT + 标准块设备（经 FTL） | ✅        | U 盘、SD 卡、eMMC、SSD    |
| FAT + 裸 NAND（无 FTL）    | ❌        | FAT 不知道坏块、不会跳过  |
| YAFFS2 + 裸 NAND           | ✅        | 嵌入式 Linux 早期方案     |
| YAFFS2 + 标准块设备        | ❌/无意义 | YAFFS2 设计就是绕过块设备 |
| UBIFS + UBI + 裸 NAND      | ✅        | 现代嵌入式 Linux          |
| ext4 + 裸 NAND             | ❌        | ext4 不处理坏块，会崩溃   |
| ext4 + FTL 块设备          | ✅        | 某些嵌入式方案            |

### 3.6 常见问题解答

**"上层不都是一样的吗？"**

**接口看起来一样（`open`/`read`/`write`），但底层实现完全不同。** 裸 NAND 专用文件系统需要调用 `nand_read_page`/`nand_program_page`/`nand_erase_block`，而 FAT 调用的是 `read_sector`/`write_sector`（标准块设备接口）。

**"文件系统有好几种，对于存储介质来说哪一种都可以吗？"**

**绝对不是。** 关键区分：

| 场景                  | 可用文件系统                | 不可用                    |
| :-------------------- | :-------------------------- | :------------------------ |
| U 盘（内部有 FTL）    | FAT/exFAT/NTFS              | YAFFS2/UBIFS（无意义）    |
| 裸 NAND（无 FTL）     | YAFFS2/UBIFS/JFFS2/LittleFS | FAT/NTFS/ext4（直接崩溃） |
| 裸 NAND + 外部 FTL 层 | FAT/exFAT（通过 FTL）       | —                         |
| NOR Flash             | JFFS2/LittleFS/裸跑         | YAFFS2（为 NAND 设计）    |

**"怎么解释 U 盘格式化时可以选文件系统？"**

因为 U 盘的**内部控制器已经完成了 FTL**，对外是标准块设备。PC 格式化工具面对的是一个"虚拟硬盘"，选 FAT 还是 exFAT 只是**在这个虚拟硬盘上创建不同的元数据结构**。U 盘控制器不管这些，它只负责把逻辑块号（LBA）翻译成 NAND 物理地址。

### 3.7 GD5F4GM7 场景分析

`nandflash_ftl` 方案：

```
SPI NAND (GD5F4GM7)
    ↓
FTL 层（系统中间件，如 nandflash_ftl）
    ↓
注册为块设备
    ↓
f_format(root, "fat", 0)  ← 在这里 FAT 被"安装"
    ↓
应用通过文件路径访问
```

这里的关键是：**FTL 层把裸 NAND 的物理特性（坏块、擦写不对称）封装掉了**，向上提供标准块设备接口。所以 FAT 可以工作——FAT 以为自己在操作"完美硬盘"，实际是 FTL 在处理 NAND 的复杂物理操作。

如果**去掉 FTL**，直接让 FAT 操作裸 NAND：

- FAT 的 FAT 表（频繁更新）会反复擦写同一个物理块
- 该块很快超过 P/E 寿命损坏
- FAT 不知道坏块存在，继续访问 → 数据错误/丢失
- 没有 ECC 协同，位翻转导致文件系统结构损坏

### 3.8 小结

> **文件系统不是"随便选"的，必须与底层介质的暴露方式匹配。裸 NAND 的物理缺陷（坏块、磨损、擦写不对称）决定了它要么配合专用文件系统（YAFFS2/UBIFS/LittleFS），要么必须先经过 FTL 封装成标准块设备后才能使用通用文件系统（FAT/ext4）。U 盘能格式化成多种文件系统，恰恰是因为内部 FTL 已经完成了介质适配，对外隐藏了 NAND 的真实特性。**

## 四、存储介质与文件系统的关系

### 4.1 核心结论与分层架构

#### 4.1.1 完全解耦的两个层次

**存储介质（如 NAND Flash、NOR Flash、SD 卡、SSD、硬盘等）本身不自带任何文件系统。** 文件系统是**纯软件层面**的数据组织规则，是写在存储介质上的"元数据结构"，而非介质本身的物理属性。

**图书馆类比：**

- **存储介质** = 空书架（只提供放书的空间）
- **文件系统** = 图书分类法（杜威十进制、中图法等）+ 目录卡片 + 借阅规则
- **FTL（Flash Translation Layer）** = 图书管理员（负责把分类号映射到实际书架位置，处理损坏的书架隔板）

#### 4.1.2 分层架构详解

```
┌─────────────────────────────────────────┐
│  应用层 (Applications)                    │
│  —— 用户直接交互：打开文件、保存照片等        │
├─────────────────────────────────────────┤
│  文件系统层 (File System)                 │
│  —— FAT/exFAT/NTFS/ext4/YAFFS2 等         │
│  —— 管理：文件名、目录树、权限、簇分配表      │
├─────────────────────────────────────────┤
│  块设备层 / FTL 层 (Block Device / FTL)    │
│  —— 逻辑块到物理块的映射、磨损均衡、坏块管理   │
│  —— 注意：FTL 位置因产品形态而异（见下文）    │
├─────────────────────────────────────────┤
│  驱动层 (Driver)                          │
│  —— SPI/QSPI/NAND Ctrl/SATA/NVMe 控制器    │
├─────────────────────────────────────────┤
│  物理介质层 (Physical Media)               │
│  —— NAND Flash / NOR Flash / 磁性盘片 等   │
│  —— 只认识：地址、字节、0 和 1              │
└─────────────────────────────────────────┘
```

### 4.2 存储介质的"无知性"：以 SPI NAND 为例

#### 4.2.1 芯片的物理视角

以 **GD5F4GM7 xF 型号**（4Gb SPI NAND，4KB+256B Page）为例，芯片的物理视角只有：

| 概念                         | 物理含义                   | 芯片是否理解"文件" |
| :--------------------------- | :------------------------- | :----------------- |
| **Block（块）**              | 擦除最小单位，256KB+16KB   | ❌ 不认识           |
| **Page（页）**               | 读写最小单位，4KB+256B     | ❌ 不认识           |
| **Column Address（列地址）** | 页内字节偏移 0~4607        | ❌ 不认识           |
| **Row Address（行地址）**    | Block + Page 组合地址      | ❌ 不认识           |

芯片支持的命令集：

- `13h` — Page Read to Cache（把一页数据从 Flash 阵列读到缓存）
- `02h` / `32h` — Program Load / x4（把数据加载到缓存）
- `10h` — Program Execute（把缓存数据写入 Flash 阵列）
- `D8h` — Block Erase（擦除一个块）

**这些命令完全不涉及任何"文件"、"目录"、"文件名"的概念。** 芯片只是忠实地执行：**在指定地址写入指定数据、从指定地址读出指定数据、擦除指定块**。

#### 4.2.2 文件系统是什么？

文件系统（以 FAT 为例）是**通过底层命令写入介质的一串普通字节**，介质本身并不知道这些字节的特殊含义：

| FAT 组件              | 物理存在形式                           | 在 NAND 中的实际存储       |
| :-------------------- | :------------------------------------- | :------------------------- |
| **MBR / 分区表**      | 描述磁盘分区布局                       | Block 0, Page 0 的若干字节 |
| **Boot Sector / BPB** | 文件系统元信息（簇大小、FAT 表位置等） | 紧随 MBR 之后的页          |
| **FAT 表**            | 簇链映射（文件碎片链接关系）           | 连续若干页/块              |
| **Root Directory**    | 根目录项（文件名、大小、起始簇等）     | 固定区域                   |
| **Data Area**         | 实际文件数据（文本、图片、代码等）     | 剩余空间                   |

**关键认知**：FAT 表的内容和一张 JPEG 照片，在 NAND 芯片看来**完全等价**——都是"某页某列的若干字节"。芯片不区分"这是文件系统元数据"还是"这是用户数据"。

### 4.3 OOB（Spare Area）详解

以 GD5F4GM7 xF 型号（4KB+256B）为例，一个 Page 的完整结构：

```
┌─────────────────────────────────────┬─────────────────────┐
│          Data Area (Main Area)       │   Spare Area (OOB)  │
│              4096 字节               │      256 字节        │
├─────────────────────────────────────┼─────────────────────┤
│  用户数据（文件内容、元数据等）       │  ECC校验码          │
│                                     │  坏块标记           │
│                                     │  文件系统元数据      │
│                                     │  （如YAFFS2页头）   │
└─────────────────────────────────────┴─────────────────────┘
```

#### 4.3.1 OOB 各区域作用

| 区域 | 位置（典型） | 作用 |
|------|-------------|------|
| **坏块标记（BBM）** | Spare Area 第 1 或第 2 个字节 | 出厂时若为 0xFF 表示好块，非 0xFF 表示坏块 |
| **ECC 校验码** | 按 ECC 算法分配 | 纠正数据位的随机翻转 |
| **文件系统元数据** | 剩余空间 | 如 YAFFS2 的文件头、JFFS2 的节点头 |

#### 4.3.2 坏块判定标准

- **出厂坏块**：每个 Block 的第 1 个 Page 和第 2 个 Page 的 Spare Area 第一个字节，若不为 0xFF 则标记为坏块
- **使用中坏块**：擦除或编程操作返回失败状态，或 ECC 无法纠正的错误
- **坏块一经标记，不再用于存储数据，由 FTL 管理替换为备用块**

### 4.4 FTL 技术机制

#### 4.4.1 FTL 的定位

FTL（Flash Translation Layer）是 NAND Flash 场景中必不可少的中间层，负责解决 NAND 的物理特性限制：

| NAND 物理限制                        | FTL 的应对策略                               |
| :----------------------------------- | :------------------------------------------- |
| 写入前必须擦除（Erase Before Write） | 逻辑写入 → 映射到新物理页 → 标记旧页为无效   |
| 块擦除次数有限（SLC ~80K，MLC ~3K-10K，TLC ~1K-3K，QLC 仅几百次） | 磨损均衡（Wear Leveling），均匀分配擦写      |
| 出厂及使用中产生坏块                 | 坏块管理（Bad Block Management），替换备用块 |
| 页大小 ≠ 传统扇区大小（512B/4KB）    | 逻辑块到物理页的地址映射                     |

#### 4.4.2 逻辑-物理映射（L2P Table）

- FTL 维护一张逻辑块号到物理块号的映射表
- **页级映射（Page-level）**：每个逻辑页映射到一个物理页，灵活性高但映射表大
- **块级映射（Block-level）**：逻辑块映射到物理块，映射表小但易产生碎片
- **混合映射**：常用方案，结合两者优势

#### 4.4.3 磨损均衡分类

| 类型 | 原理 | 效果 |
|------|------|------|
| **动态磨损均衡** | 新数据优先写入磨损较少的空闲块 | 解决写入热点的磨损不均 |
| **静态磨损均衡** | 定期将冷数据（少更新）从低磨损块迁移到高磨损块，让低磨损块参与写入 | 解决静态数据的长期不均 |

#### 4.4.4 垃圾回收（Garbage Collection）

- 当 NAND 中无效页（被更新后标记为废弃的页）积累到一定阈值时，FTL 触发 GC
- GC 将有效页从待回收块复制到新块，然后擦除原块释放空间
- GC 策略影响性能和写入放大（Write Amplification）因子

#### 4.4.5 FTL 的位置因产品形态不同而变化

| 产品形态                | FTL 位置                  | 对外接口         | 例子                 |
| :---------------------- | :------------------------ | :--------------- | :------------------- |
| **Raw NAND 芯片**       | **外部软件实现**          | 需自定义驱动     | GD5F4GM7 直连 MCU    |
| **eMMC / SD 卡**        | **芯片内部控制器**        | 标准块设备接口   | 手机存储卡           |
| **SSD**                 | **主控芯片内部**          | SATA/NVMe        | 电脑固态硬盘         |
| **U 盘**                | **USB 桥接 + Flash 主控** | USB Mass Storage | 普通 U 盘            |
| **SPI NAND + 系统方案** | **系统中间件层**          | 块设备抽象       | `nandflash_ftl` 注册 |

**重要**：即使 FTL 在芯片内部（如 eMMC），文件系统仍然是**外部写入的**——eMMC 控制器只保证"逻辑块可靠存储"，FAT/exFAT 还是由主机（手机、相机、PC）格式化时写入。

### 4.5 格式化：文件系统的"创建"过程

格式化不是"激活芯片隐藏功能"，而是**向空白介质写入文件系统元数据结构**：

```
f_format(root_path, "fat", 0)  的执行过程：
    │
    ├── 1. 通过 dev_manager 获取块设备句柄
    │      （底层可能是 nandflash_ftl → SPI NAND 驱动）
    │
    ├── 2. 在 Block 0 写入 MBR / 分区表
    │
    ├── 3. 在后续块写入 Boot Sector（BPB 参数）
    │      —— 簇大小、FAT 表位置、根目录位置、总扇区数等
    │
    ├── 4. 初始化 FAT 表（清 0，标记坏簇）
    │
    ├── 5. 清空 Root Directory
    │
    └── 6. 标记 Data Area 为空白
```

**USB 接 PC 格式化**只是另一种触发上述流程的方式：

- PC 通过 USB Mass Storage 协议发送 SCSI 格式化命令
- 设备端固件接收命令 → 调用相同的 `f_format` 逻辑
- 或者设备作为纯透传，由 PC 端工具直接写物理块

**设备端自主格式化**（不依赖 USB 连接）完全可行：

```c
// 注册 FTL 块设备
dev_manager_add("nandflash_ftl");

// 获取根路径
const char *root = dev_manager_get_root_path_by_logo("nandflash_ftl");

// 直接在设备端格式化
f_format(root, "fat", 0);
```

### 4.6 掉电安全（Power-loss Safety）

嵌入式系统频繁上下电，掉电保护是核心关切：

| 文件系统 | 掉电安全机制 | 说明 |
|----------|-------------|------|
| **FAT16/32** | ❌ 无保护 | 掉电时正在更新 FAT 表或目录项，极易导致文件系统损坏 |
| **ext3/ext4** | ✅ Journal | 日志机制，掉电后通过日志恢复一致性 |
| **JFFS2** | ✅ 日志结构 | 顺序写入，天然抗掉电 |
| **UBIFS** | ✅ 日志 + WAL | 配合 UBI 层的写前日志 |
| **LittleFS** | ✅ Copy-on-write | 原子性更新，掉电安全是设计核心 |

#### 4.6.1 掉电保护建议

- **裸 NAND 场景**：优先选用 LittleFS 或 UBIFS
- **需要 FAT 兼容性的场景**：确保关键写操作后调用 `sync`，配合超级电容/UPS 缓冲
- **eMMC/SSD 场景**：依赖内部 PLP（Power Loss Protection）机制，高端产品内置超级电容

#### 4.6.2 关键概念

- **Write Atomicity（写原子性）**：一个操作要么完全完成，要么完全不发生
- **Double-Write**：某些系统采用双写策略确保元数据完整性
- **Sync/Flush**：强制将缓存数据写入物理介质

### 4.7 常见误区澄清

| 误区                                   | 正解                                            |
| :------------------------------------- | :---------------------------------------------- |
| "NAND 芯片出厂自带 FAT，插上去就能用"  | ❌ 芯片完全空白，FAT 必须由外部写入              |
| "文件系统是硬件功能，像 ECC 一样内置"  | ❌ ECC 可选内置，文件系统永远是软件              |
| "USB 格式化是必须的，否则没有文件系统" | ❌ 设备端完全可以自主格式化                      |
| "FTL 和文件系统是一回事"               | ❌ FTL 管"物理块映射"，文件系统管"文件名和目录"  |
| "坏块管理是文件系统做的"               | ❌ 坏块管理是 FTL 的职责，文件系统不感知物理坏块 |

### 4.8 小结

> **存储介质芯片只提供最基础的物理操作：读页、写页、擦块。文件系统是纯粹的上层软件规则，通过相同的物理写操作被"安装"到介质上。FTL 作为中间层，解决 NAND 特有的物理限制（坏块、磨损、擦写不对称），其位置可内可外，但文件系统始终独立于 FTL 和物理介质，是数据组织的高级抽象。**

| 层级                  | 是否属于芯片 | 是否属于软件 | 职责         |
| :-------------------- | :----------- | :----------- | :----------- |
| 物理介质（NAND 颗粒） | ✅            | ❌            | 存 0/1       |
| FTL                   | 可内可外     | 本质是算法   | 物理块管理   |
| 文件系统              | ❌            | ✅            | 数据组织规则 |
| 应用                  | ❌            | ✅            | 用户意图     |

## 五、存储介质厂商的软件交付范围

### 5.1 厂商实际交付内容

**Flash 芯片厂商通常只提供硬件 + 使用说明书（Datasheet），不会提供完整的软件栈。** 上层的 FTL、文件系统、应用框架都需要开发者自行解决或寻找第三方方案。

| 交付物                             | 是否提供              | 具体内容                                           | 用途                       |
| :--------------------------------- | :-------------------- | :------------------------------------------------- | :------------------------- |
| **芯片硬件**                       | ✅                     | SPI-NAND / NOR / eMMC / SD 等颗粒                  | 物理存储介质               |
| **Datasheet**                      | ✅                     | 电气特性、命令集、寄存器定义、AC/DC 时序、封装信息 | 硬件设计和底层驱动开发依据 |
| **应用笔记（App Note）**           | ⚠️ 部分提供            | 特定场景的使用建议（如 OTP 操作、BBM 流程）        | 参考实现流程               |
| **评估板/参考设计**                | ⚠️ 大客户/部分型号提供 | 原理图、PCB 布局、配套代码                         | 快速验证和原型开发         |
| **基础命令示例代码**               | ⚠️ 有时提供            | 基于特定 MCU 的 Page Read/Program/Erase 封装       | 驱动开发起点               |
| **FTL（Flash Translation Layer）** | ❌ **不提供**          | —                                                  | 需自行实现或第三方         |
| **文件系统（FAT/exFAT 等）**       | ❌ **不提供**          | —                                                  | 需自行移植或第三方         |
| **完整系统方案**                   | ❌ **不提供**          | —                                                  | 需自行整合或找方案商       |

### 5.2 为什么厂商不提供上层软件？

| 原因                                   | 说明                                            |
| :------------------------------------- | :---------------------------------------------- |
| **芯片厂商的核心竞争力是工艺和可靠性** | 卖的是物理存储能力，不是软件方案                |
| **上层依赖具体应用场景**               | MCU 型号、RTOS 选择、性能需求千差万别，无法统一 |
| **生态分工明确**                       | 芯片 → 模组 → 方案商 → 终端产品，每层加值       |
| **软件维护成本高**                     | 适配不同平台工作量巨大，不如专注硬件            |

### 5.3 各层级软件的实际来源

```
┌─────────────────────────────────────────┐
│  文件系统 (FAT/exFAT/LittleFS)           │  ← 开发板厂商预装 / RTOS / 开源库
├─────────────────────────────────────────┤
│  FTL (Flash Translation Layer)           │  ← 开发板厂商预装 / 方案商 / 自行实现
├─────────────────────────────────────────┤
│  块设备抽象 / dev_manager                │  ← 系统框架（开发板厂商或自研）
├─────────────────────────────────────────┤
│  SPI 控制器驱动                          │  ← MCU 原厂 HAL（开发板厂商整合）
├─────────────────────────────────────────┤
│  Flash 命令封装层                         │  ← 开发板厂商基于 Datasheet 封装
├─────────────────────────────────────────┤
│  物理芯片 (GD5F4GM7)                      │  ← Flash 芯片原厂
└─────────────────────────────────────────┘
```

### 5.4 典型开发流程

以 **GD5F4GM7** 为例，开发者需要完成的工作：

| 步骤                   | 工作内容                                              | 参考来源           |
| :--------------------- | :---------------------------------------------------- | :----------------- |
| 1. 读 Datasheet        | 理解 Page/Block 结构、命令集、时序要求                | 厂商 Datasheet     |
| 2. 配置 MCU SPI 控制器 | 设置 SPI 模式、时钟频率、CS 控制                      | MCU 原厂 HAL       |
| 3. 写底层命令封装      | 实现 `read_page()`、`program_load()`、`block_erase()` | Datasheet 时序图   |
| 4. 集成或实现 FTL      | 逻辑-物理映射、坏块管理、磨损均衡                     | 第三方库 / 自研    |
| 5. 注册块设备          | `dev_manager_add("nandflash_ftl")`                    | 系统框架           |
| 6. 格式化文件系统      | `f_format(root, "fat", 0)`                            | FAT 库（如 FatFs） |
| 7. 应用层使用          | `f_open()`、`f_write()` 等                            | 文件系统 API       |

### 5.5 例外情况

| 场景                 | 厂商额外提供               | 说明                        |
| :------------------- | :------------------------- | :-------------------------- |
| **大客户的定制项目** | 完整驱动 + 技术支持        | 量大有专门 FAE 对接         |
| **turnkey（交钥匙）方案商** | 模组级交付（含 FTL + FAT） | 如某些 eMMC/SD 模组厂商     |
| **特定生态系统绑定** | 配套 SDK                   | 如 ESP32 官方支持特定 Flash |
| **高价值工业级产品** | 延长寿命算法、加密方案     | 附加值服务                  |

但这些都是**增值服务的延伸**，不是芯片本身的标准交付物。

### 5.6 开发板厂商与芯片原厂的区别

| 对比项       | 芯片原厂（如 GigaDevice） | 开发板厂商（如正点原子、野火）        |
| :----------- | :------------------------ | :------------------------------------ |
| **目标**     | 卖芯片给所有人            | 卖"能跑起来的系统"给开发者            |
| **交付形态** | 裸芯片 / reels            | 开发板 + 例程 + 文档                  |
| **软件提供** | 极少，只有 Datasheet      | **完整 SDK**：驱动 + FTL + FAT + Demo |
| **技术支持** | 硬件规格咨询              | 代码级问题解答                        |
| **典型代码** | 无或极简 SPI 读写示例     | `f_open()` 直接可用的完整工程         |

**关键差异**：开发板厂商的核心竞争力就是**降低上手门槛**，所以他们必须把 FTL、FAT、示例代码都打包好，否则开发者买回去无法直接使用。

### 5.7 方案商（Turnkey）的角色

比开发板厂商更进一步：

| 层级 | 方案商提供                           |
| :--- | :----------------------------------- |
| 硬件 | 定制 PCB + 选型的 Flash 芯片         |
| 底层 | 适配好的 SPI/FTL 驱动                |
| 系统 | 文件系统 + 设备管理框架              |
| 应用 | 业务逻辑（如录音笔固件、词典笔系统） |
| 生产 | 量产工具、烧录流程、测试方案         |

**方案商卖给终端品牌的是"直接拿去贴牌的产品"**，所以必须全栈交付。

### 5.8 小结

> **存储介质厂商的本质是物理存储器件的制造者，不是软件方案提供商。Datasheet 是他们唯一必须交付的"使用说明书"，里面的命令时序图就是驱动开发的全部依据。从物理命令到文件系统的鸿沟，需要开发者自己或借助第三方生态来填补。**

**FTL、FAT 等上层软件一般由以下角色提供：**

| 角色                  | 提供内容                               | 典型场景         |
| :-------------------- | :------------------------------------- | :--------------- |
| **芯片原厂**          | Datasheet、硬件、有时附赠基础命令示例  | 买裸芯片自己开发 |
| **开发板/模组厂商**   | **完整驱动栈（SPI 驱动 + FTL + FAT）** | 卖可即用的开发板 |
| **方案商/集成商**     | **turnkey（交钥匙）方案（硬件 + 软件一站式）** | 交付终端产品方案 |
| **RTOS/操作系统厂商** | 文件系统组件 + 设备框架                | 生态配套         |
| **开源社区**          | FatFs、LittleFS、UBIFS 等              | 自行移植         |

## 六、附录：存储介质选型决策指南

### 6.1 选型决策表

| 需求场景 | 推荐介质 | 推荐文件系统 | 理由 |
|----------|----------|-------------|------|
| Bootloader 存储（<16MB） | NOR Flash | 裸跑/JFFS2/LittleFS | 支持 XIP，字节随机读 |
| 固件+配置存储（16MB~512MB） | SPI NAND | LittleFS/UBIFS/YAFFS2 | 性价比高，容量适中 |
| 操作系统+应用（512MB~32GB） | eMMC | ext4/FAT/exFAT | 标准块设备，集成简单 |
| 高性能大容量（>32GB） | SSD/UFS | ext4/NTFS/APFS | 高带宽，大容量 |
| 可插拔存储 | SD卡/TF卡 | FAT/exFAT | 兼容性好 |

### 6.2 多介质协同设计（以智能 IPC 为例）

```
┌─────────────────────────────────────────────────┐
│  NOR Flash (8~32MB)                             │
│  └── Bootloader + 恢复固件                       │
│  └── 文件系统：裸跑（XIP）或 JFFS2               │
├─────────────────────────────────────────────────┤
│  SPI NAND (128MB~1GB) 或 eMMC                   │
│  └── Linux内核 + 根文件系统 + 应用程序           │
│  └── 文件系统：UBIFS / LittleFS / ext4           │
├─────────────────────────────────────────────────┤
│  SD卡 / 外接SSD（可选）                          │
│  └── 录像数据、日志、用户数据                     │
│  └── 文件系统：exFAT / FAT32                     │
├─────────────────────────────────────────────────┤
│  DRAM (LPDDR)                                   │
│  └── 系统运行内存                                │
└─────────────────────────────────────────────────┘
```

### 6.3 选型关键考量因素

| 考量因素 | 说明 |
|----------|------|
| **容量需求** | 确定最小和最大容量范围，预留 20% 余量 |
| **读写性能** | 顺序读/写 vs 随机读/写的不同要求 |
| **寿命要求** | 根据写入频率估算 P/E 寿命是否满足 |
| **掉电安全** | 系统是否频繁上下电，是否需要掉电保护 |
| **成本预算** | 芯片成本 + 控制器成本 + 开发成本综合评估 |
| **供应保障** | 厂商供货稳定性、交期、替代料可行性 |
| **生态支持** | 驱动、FTL、文件系统的成熟度和社区活跃度 |
